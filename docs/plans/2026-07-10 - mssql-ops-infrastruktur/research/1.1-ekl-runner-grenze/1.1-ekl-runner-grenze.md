# EKL-Migrations-Runner (excel_ekl) — Grenzanalyse für grate/RoboticoOps

> Quelle: Opus-Agent „ekl-runner-boundary", Session 2026-07-09, Read-only-Audit von `/home/lukas/WebStorm/excel_ekl`.
> **Zentraler Befund: Der EKL-Runner ist NICHT sauber auf `RoboticoEKL` beschränkt** — er bewohnt auch `CustomWorkflows` und schreibt in `dbo.tWorkflow`/`tWorkflowAktion`.

## 1. Runner-Mechanik

- **TypeScript-CLI** (`scripts/jtl-migrate.ts`, Commander; npm `migrate:jtl`), Befehle `status | up | validate | create | backup`. **Kein Auto-Run beim App-Start** — Boot prüft nur (Health-Plugin, Policy `jtl: 'error'`: pending Migrationen = fataler Startfehler mit Anleitung; Auto-Apply nur bei explizitem `'auto-migrate'`).
- Aktiver Pfad: `backend/src/infrastructure/migration/jtl/JtlMigrationOrchestrator.ts` (älterer `MssqlMigrationAdapter.ts` deprecated). Treiber `mssql`/tedious; `up.sql` wird an `^GO$` gesplittet, batchweise ausgeführt — **keine übergreifende Transaktion**; jede Datei bringt eigenes `BEGIN TRAN`+`IF NOT EXISTS`-Idempotenz mit.
- **Layout:** `backend/migrations/jtl/NNN_snake_case/` mit `up.sql`, `verify.sql`, optional `pre-check.sql`, optional `data-migration.ts`; zentrale `manifest.json`. Integer-Versionen, Stand 026 im main-Tree.
- **Journal `RoboticoEKL.tMigrationHistory`:** `nVersion`, `cFileName`, `cChecksum` (SHA-256), `bSuccess`, `cErrorMessage`, `cAppliedBy`; UNIQUE(nVersion)+UNIQUE(cFileName); Schreiben = MERGE auf nVersion (eine Zeile pro Version, letzter Versuch). „Applied" = `bSuccess=1`; FAILED zählt als pending → Re-Run. **`cChecksum` wird geschrieben, aber NIE geprüft** — nachträglich editierte, angewandte up.sql wird nicht erkannt.
- **Locking:** `sp_getapplock`, Resource **`RoboticoEKL_Migration`**, Exclusive, Transaction-Owner, Timeout 30 min. Advisory (serialisiert nur EKL-Läufe).
- **Backup:** `up` erzwingt Voll-Backup der gesamten Ziel-DB vor jeder Migration. Kein `down.sql` — Recovery = Restore.

## 2. Scope-Grenze — greift über RoboticoEKL hinaus

CREATE-Ziele: 63× RoboticoEKL, **4× CustomWorkflows**. Keine dbo-Tabellen-DDL, aber:

- **Legt `CustomWorkflows`-Schema an** (016, 023) und erstellt darin `spCMArtikel`, `spCMArtikelNeu` — **dasselbe Schema, das JTL-Robotico bespielt**.
- **Konsumiert die CW-Registrierungs-Infrastruktur von JTL-Robotico:** `CustomWorkflows._CheckAction`, `._SetActionDisplayName`, `vCustomAction` (guarded, sonst nur PRINT-Warnung). Diese gehören JTL-Robotico; EKL erwartet sie als vorhanden.
- **Schreibt in `dbo.tWorkflow`/`tWorkflowAktion`** (017, 023, 024): legt aktive JTL-Workflows an; 017 deaktiviert bestehende JTL-Workflows (`UPDATE dbo.tWorkflow SET nTyp=2`). Idempotenz per `cName`-Marker („EKL …").
- **Steady-State-DML in dbo** (zur Laufzeit via SPs): tArtikel, tArtikelLabel, tArtikelShop, tShopUeberverkaeufeMoeglich, ebay_item, ebay_logbuch u.a.
- **FK-Kopplungen auf JTL-dbo:** RoboticoEKL-Tabellen referenzieren dbo.tArtikel/tLabel/tWarengruppe/tBenutzer (teils ON DELETE CASCADE).
- **Keine Grants/Logins/User/Server-Objekte** — der Runner erzeugt keine Prinzipale.

## 3. Ziel-DB-Konfiguration

- Env-getrieben (node-config-Cascade, `--environment=<name>`, TTY-Bestätigung für Prod). Ein Lauf = genau eine DB. Environments: test-docker, test-mssql, test-production/test-staging (**VM-SQL-Test-1**), production (**vm-sql2**, eazybusiness).
- **Keine Referenz auf `eazybusiness_tm2/3/4`** — der EKL-Runner läuft nie gegen die Testmandanten-Klone, nur gegen die jeweilige Haupt-eazybusiness (Beleg: Prod-Lauf 2026-07-02, v1–v24 auf vm-sql2).

## 4. Kollisionsflächen mit grate/RoboticoOps

- **HARTE Kollision: `CustomWorkflows`.** Beide Ketten schreiben in dasselbe Schema. grate darf `spCMArtikel`/`spCMArtikelNeu` und EKL-Zeilen in `dbo.tWorkflow` (cName `EKL …`) nie anfassen. Umgekehrt: Wird die Registrierungs-Infra (`_CheckAction` etc.) umbenannt/entfernt, brechen EKL 016/017/023.
- **Journal wandert mit dem Klon:** EKL verkraftet das (Idempotenz + bSuccess-Skip, keine Checksum-Prüfung). **grate prüft Hashes** → One-time-Skripte nach Anwendung nie editieren (Standard-Disziplin; `--warnononetimescriptchanges` als Notventil); Anytime-Hash-Mismatch ist harmlos (Re-Run).
- **Getrennte Locks/Journale:** grate-Journal in eigenem Schema (Robotico bzw. RoboticoOps), kein Konflikt mit `RoboticoEKL_Migration`-Applock.
- **RoboticoOps:** für den EKL-Runner unsichtbar (kennt nur seine eine DB) — null Berührung.

## Verbindliche Grenzziehung

**EKL-Runner-Eigentum (grate NIE anfassen):**
- Schema `RoboticoEKL.*` komplett inkl. `tMigrationHistory` + Applock `RoboticoEKL_Migration`.
- In CustomWorkflows: `spCMArtikel`, `spCMArtikelNeu`.
- `dbo.tWorkflow`/`tWorkflowAktion`-Zeilen mit cName `EKL …`.
- EKL-getriebener Zustand in dbo (tArtikelLabel etc.).

**JTL-Robotico/grate-Eigentum (EKL konsumiert, verlässt sich darauf):**
- Das CW-Registrierungs-Framework: `_CheckAction`, `_SetActionDisplayName`, `vCustomAction` — **Namen/Signaturen stabil halten** (API für EKL!).
- `Robotico.*` (EKL-012 liest z. B. `Robotico.fnEscapedCSVParseLine` — auch das ist konsumierte API).
- Die neue RoboticoOps-DB.

**Geteilte Zone (nur additiv, koordiniert):** Schema `CustomWorkflows` als Container ist gemeinsam bewohnt — jede Seite nur ihre eigenen benannten Objekte, kein DROP SCHEMA, kein Löschen fremder Objekte; idempotente `IF NOT EXISTS`-Schema-Erzeugung beidseitig verträglich. `dbo` (JTL-Vendor): beide nur nach denselben Idempotenz-/ID-by-Name-Regeln.

## Übertragbare Lessons

1. **Eigenes Schema + eigener Lock-/Journal-Namensraum für grate** — einzige verbleibende Kollisionsfläche ist das additiv geteilte CustomWorkflows.
2. **Klon-Robustheit ist Design-Zwang:** Journal wandert mit jedem Klon; One-time-Skripte nach Anwendung nie editieren; Anytime-Re-Runs müssen folgenlos sein.
3. **GO-Split ohne Gesamt-Transaktion → Objekt-granulare Idempotenz erzwingen** (grate: `--transaction` nutzen, trotzdem idempotent schreiben).
4. **Keine hardcodierten JTL-IDs — immer by-Name auflösen**; Pre-Checks als hard FAIL, nicht WARN (schärfste EKL-Prod-Risiken kamen aus Test-DB-IDs).
5. **Deploy strikt manuell, mit Backup, nie Auto-Apply** — EKL-Posten übernehmen (CLI + erzwungenes Backup + Prod-Bestätigung).
